Abstract

Digital fitness ecosystems require high data persistence, seamless layout adaptations, and centralized administration to ensure high user retention. This project report presents the system design and implementation of FitLife, a responsive web application for personalized fitness plan distribution and athlete progress monitoring.

The previous generation of local workout web apps was heavily limited by local device storage, resulting in immediate progress logs and active plan data loss upon browser refreshes, cache clearing, or device switching. To address these database persistence limitations, the proposed FitLife platform utilizes a serverless architecture integrating a client-side interface with Google Cloud Firestore. By using email addresses as unique identifiers, the application automatically loads custom workout splits, water logs, weight progression curves, and query tickets.

Additionally, a secure, isolated administration dashboard (admin.html) was developed to enable administrators to search user profiles, verify payment transaction references (UTR), approve paid tier unlocks, and change plan states dynamically. The frontend leverages Vanilla CSS for a charcoal-and-red sports branding style, HTML5 for semantic structuring, and JavaScript ES6 for state sync. Testing and validation demonstrate robust state recovery, secure Firestore reads/writes, and zero data leakage across athlete accounts.

1. Introduction

1.1 Overview of the Project

In recent years, the intersection of mobile technology, cloud database solutions, and web frameworks has transformed the health and fitness industry. Users expect modern interfaces to track their daily activities, hydration levels, body weight progress, and exercise form dynamically. Fitness platforms must balance lightweight client execution with robust real-time updates to manage user data across multiple devices.

The project, entitled "FitLife Fitness Web Platform", is a cloud-integrated web application designed to manage personalized workout splits, diet blueprints, and athlete progress records. Styled under a high-contrast black-and-red athletic theme, the platform serves two primary interfaces: an athlete workspace dashboard and a secure administrative panel. The backend utilizes Google Cloud Firestore to store and synchronize records in real-time, removing the limitations of local device-only session caches.

1.2 Objectives of the Project

The core development objectives for the FitLife system are as follows:

  • Persistent Database Architecture: Establish Cloud Firestore as the centralized repository to replace unstable client-side browser caching, ensuring data integrity.
  • Session Restoration & Autologin: Implement mechanisms using localStorage cache and Firestore queries to detect returning users during browser page refreshes or device logins, loading their active plan layouts and historical metrics.
  • Payment Workflow Verification: Provide standard and premium users with payment options via QR code or copyable UPI IDs. Develop text validation modules to secure 12-digit transaction references.
  • Centralized Administrative Moderation: Engineer an isolated, passcode-protected administrative dashboard to authorize paid tier changes, verify billing UTR values, analyze query histories, and delete accounts.
  • Responsive Interactive Dashboard: Formulate a layout offering daily workout checklist updates, log progression charts, diet selectors, and form guide video databases.

2. Software Development Life Cycle (SDLC)

2.1 Overview of SDLC

A Software Development Life Cycle (SDLC) is a structured software engineering framework used by development teams to design, code, test, deploy, and maintain software programs systematically. Following an SDLC ensures that products meet specified quality metrics, budget limitations, and client requirements. The cycle typically comprises six major phases: Requirement Gathering, System Analysis, System Design, Coding/Development, Testing, and Deployment & Maintenance.

2.2 SDLC Model Selected

For the FitLife Fitness Web Platform, an Agile Prototyping Model was selected. Unlike the traditional sequential Waterfall approach, this iterative model combines the advantages of prototyping (immediate UI feedback) with the rapid increments of Agile software development.

This model was chosen due to the following project conditions:

  • Iterative Development of Security and DB Systems: The migration from LocalStorage data structures to a serverless Firestore database model required multiple database design iterations.
  • User Interface Adjustments: The high-contrast sports theme CSS styling and administrative dossier drawer transitions required constant micro-adjustments and layout feedback.
  • Rapid Feature Unlocking: Features such as the water tracker, query desk, and multi-day workout split selectors were implemented as distinct, self-contained increments that were added to the dashboard dynamically.

3. System Study and Analysis

3.1 Problem Statement

Modern wellness tracking applications must maintain state across page reloads and browser transitions. The initial generation of the FitLife web application stored all user statistics (such as water intake totals, weight progressions, and checked exercises list) in the client's browser local cache (LocalStorage).

This architecture suffered from critical flaws:

  1. Volatile Data Cache: Clearing browser history, cookies, or accessing the website via private browsing modes (Incognito) wiped out the athlete's workout progress history.
  2. Device Dependency: An athlete could not access their standard workout plan or log weight on a mobile device if they had originally registered on a desktop computer.
  3. Billing Bypass & Security Vulnerability: The client-side logic was vulnerable to browser console manipulation, allowing users to unlock premium and standard sections without paying.
  4. Administrative Blindspot: Coaches and system administrators had no central mechanism to verify payment transaction codes, delete user logs, or adjust user access.

3.2 Existing System Description

The existing system was a static, client-side-only web platform. It used basic HTML forms to collect names, emails, and plan preferences. If a paid plan (Standard) was selected, the system prompted the user with standard billing inputs but did not save or verify the transaction ID. Progress was written to browser cookies or LocalStorage. There was no database connection, backend service, or admin page.

3.3 Limitations of Existing System

  • No real-time data backup.
  • Frequent data loss on browser refresh or cookie cleanup.
  • No multi-device access.
  • Zero administrator control.
  • Vulnerable to local cache spoofing.

3.4 Proposed System

The proposed FitLife Fitness Platform solves these issues by connecting the client-side interface to a cloud-hosted Firebase Firestore Database.

Key enhancements include:

  • Real-time Cloud Sync: All hydration counts, completed exercise checklists, weight lists, and trainer queries are synced to Cloud Firestore documents under the user's email document ID.
  • Unique Identity Session Engine: Email is used as the unique identifier. The onboarding system checks for returning profiles, restoring layouts and progress values.
  • Protected Administrator Dashboard: A secure admin.html portal allows administrators to filter records, review UTR details, verify billing, adjust plan tiers, and manage database records.
  • Secure Rules Verification: Firestore security rules ensure that only authenticated updates occur, protecting user documents from external database injections.

3.5 Feasibility Study

Technical Feasibility

The system is built using HTML5, CSS3, JavaScript ES6, and Firebase SDK libraries. Because Firebase functions as a serverless Backend-as-a-Service (BaaS), the system does not require dedicated node servers, making client execution fast and technical deployment straightforward.

Economic Feasibility

Google Firebase offers a generous free tier (Spark Plan) allowing 50,000 document reads, 20,000 document writes, and 20,000 document deletes per day. This makes the system economically feasible, as there are no backend hosting costs during the initial deployment phase.

Operational Feasibility

The athlete interface is designed for intuitive mobile and desktop use. The administration portal simplifies back-office verification tasks by using visual status badges, instant action triggers (Approve, Reject), and details drawers.

4. System Requirements

4.1 Functional Requirements

Functional requirements define the core functions the system must perform:

  • User Onboarding and Plan Selection: The system must accept name, email, and plan tier parameters. Paid plan requests must route users to the payment layout.
  • Email Verification and Profile Loading: The system must check Firestore for existing email records. If found, it must load the user's layout, progress, and logs, bypassing payment if already verified.
  • Progress Synchronization: Whenever a user logs water, weight, or checks off a workout, the client must trigger an incremental database update without reloading the page.
  • Payment Verification Logic: The payment portal must enforce a 12-digit numeric constraint for transaction reference inputs.
  • Admin Records Management: The admin panel must load database profiles, filter by name/email/plan, update status, and delete records.

4.2 Non-Functional Requirements

  • Performance: Data syncing and loading operations must complete within 2 seconds under ordinary mobile internet connections.
  • Data Security: Access to user collections must be filtered by matching document path IDs.
  • Usability: The client UI must adapt to mobile, tablet, and desktop viewports using CSS media queries.
  • Reliability: If the database is unreachable, the system must fall back to local storage and sync changes later.

const firebaseConfig = {

let db = null;
try {
  if (typeof firebase !== 'undefined' && firebaseConfig.projectId && !firebaseConfig.projectId.includes("YOUR_PROJECT_ID")) {
    firebase.initializeApp(firebaseConfig);
    db = firebase.firestore();
    console.log("Firebase Firestore initialized successfully!");
  } else {

  }
} catch (error) {
  console.error("Firebase Firestore initialization failed. LocalStorage mode:", error);

// 1. SAVE USER DATA
window.dbSaveUser = async function(email, userData) {
  if (!email) return false;
  
  const cleanEmail = email.toLowerCase().trim();
  const dateStr = new Date().toLocaleDateString();

  // Create document update object containing only defined values
  const docData = {
    userId: cleanEmail,
    email: cleanEmail,
    lastUpdated: new Date().toISOString()
  };

  if (userData.name !== undefined) docData.name = userData.name;
  
  if (userData.plan !== undefined) {
    docData.plan = userData.plan;
    docData.activePlan = userData.plan;
  }
  if (userData.activePlan !== undefined) {
    docData.activePlan = userData.activePlan;
  }

  if (userData.txnRefId !== undefined || userData.transactionId !== undefined) {
    const txnVal = userData.txnRefId || userData.transactionId || "N/A";
    docData.txnRefId = txnVal;
    docData.transactionId = txnVal;
  }

  if (userData.paymentStatus !== undefined) {
    docData.paymentStatus = userData.paymentStatus;
  } else if (userData.plan !== undefined) {
    docData.paymentStatus = userData.plan === "Free" ? "Verified" : "Pending Verification";
  }

  if (userData.registrationDate !== undefined) docData.registrationDate = userData.registrationDate;
  if (userData.waterCount !== undefined) docData.waterCount = userData.waterCount;
  if (userData.waterTarget !== undefined) docData.waterTarget = userData.waterTarget;
  if (userData.completedExercisesCount !== undefined) docData.completedExercisesCount = userData.completedExercisesCount;
  if (userData.totalExercisesCount !== undefined) docData.totalExercisesCount = userData.totalExercisesCount;
  if (userData.completedExercisesList !== undefined) docData.completedExercisesList = userData.completedExercisesList;
  if (userData.weightLogs !== undefined) docData.weightLogs = userData.weightLogs;
  if (userData.queries !== undefined) docData.queries = userData.queries;
  if (userData.fitnessGoal !== undefined) docData.fitnessGoal = userData.fitnessGoal;
  if (userData.workoutProgress !== undefined) docData.workoutProgress = userData.workoutProgress;
  if (userData.lastLogin !== undefined) docData.lastLogin = userData.lastLogin;

  // Sync to local client by merging
  let mergedData = {};
  const localData = localStorage.getItem("fitlife_current_user");
  if (localData) {
    try {
      const parsed = JSON.parse(localData);
      if (parsed && parsed.email === cleanEmail) {
        mergedData = parsed;
      }
    } catch (e) {
      console.error("Error parsing local user session:", e);
    }
  }
  Object.assign(mergedData, docData);
  localStorage.setItem("fitlife_current_user", JSON.stringify(mergedData));

  if (db) {
    try {
      await db.collection("users").doc(cleanEmail).set(docData, { merge: true });
      console.log("User successfully written to Firestore:", cleanEmail, docData);
      return true;
    } catch (e) {
      console.error("Error writing user to Firestore:", e);
      return false;
    }
  }
  return true;
};

// app.js - Profile Onboarding and Progress Synchronization Logic

// 1. Returning User Auto-Login on DOM Load
const localUserStr = localStorage.getItem("fitlife_current_user");
if (localUserStr) {
  try {
    const localUser = JSON.parse(localUserStr);
    if (localUser && localUser.email) {
      setTimeout(async () => {
        if (typeof window.dbGetUser === 'function') {
          const existingUser = await window.dbGetUser(localUser.email);
          if (existingUser) {
            if (existingUser.paymentStatus === "Rejected") {
              alert("Your Standard Plan payment verification was rejected by admin. Loading Free Plan instead.");
              existingUser.plan = "Free";
              existingUser.activePlan = "Free";
            }
            
            // Restore session variables
            window.registeredName = existingUser.name;
            window.registeredEmail = existingUser.email;
            window.registeredPlan = existingUser.activePlan || existingUser.plan || "Free";
            window.registrationDate = existingUser.registrationDate;
            window.txnRefId = existingUser.txnRefId || existingUser.transactionId || "N/A";
            window.userWaterCount = existingUser.waterCount || 0;
            window.userWaterTarget = existingUser.waterTarget || 8;
            window.completedExercisesList = existingUser.completedExercisesList || [];
            window.selectedFitnessGoal = existingUser.fitnessGoal || "gain";
            
            // Initialize Dashboard View
            unlockDashboardUI(window.registeredName, window.registeredEmail, window.registeredPlan);
          }
        }
      }, 150);
    }
  } catch (e) {
    console.error("Error reading local user session:", e);
  }
}

// 2. Incremental Database Progress Syncer
window.syncUserProgress = async function() {
  const email = window.registeredEmail;
  if (!email) return;

  const totalOverall = window.registeredPlan === "Free" ? 4 : 33;
  const completedOverall = window.completedExercisesList ? window.completedExercisesList.length : 0;

  const progressUpdate = {
    waterCount: window.userWaterCount || 0,
    waterTarget: window.userWaterTarget || 8,
    completedExercisesCount: completedOverall,
    totalExercisesCount: totalOverall,
    completedExercisesList: window.completedExercisesList || [],
    weightLogs: window.weightLogs || [],
    queries: window.trainerQueries || []
  };

  if (typeof window.dbSaveUser === 'function') {
    await window.dbSaveUser(email, progressUpdate);
  } else {
    // Local fallback
    let localData = localStorage.getItem("fitlife_current_user");
    if (localData) {
      let parsed = JSON.parse(localData);
      Object.assign(parsed, progressUpdate);
      localStorage.setItem("fitlife_current_user", JSON.stringify(parsed));
    }
  }
};
